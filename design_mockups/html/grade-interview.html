Interview Question

Design a URL Shortener

Functional Requirements
  • Users can submit a long URL and receive a shortened alias
  • Visiting the short URL redirects to the original URL
  • Users can optionally set a custom alias
  • Links can have an optional expiry date
  • Track click counts per shortened URL
Non-Functional Requirements
  • System must handle 100M URL shortenings per day
  • Redirect latency must be <10ms at the 99th percentile
  • High availability — 99.99% uptime
  • URLs must remain valid for at least 5 years
  • System should be horizontally scalable
2

Candidate's Answer

Reasoning only — v0
I'd start by estimating scale: 100M writes/day is ~1,200 writes/sec and assuming a 100:1 read-write ratio, ~120,000 reads/sec. For the core service, I'd use a stateless app layer behind a load balancer. Short codes can be generated with Base62 encoding of an auto-incremented ID, which avoids hash collisions entirely. Alternatively, we could use a distributed ID generator like Snowflake if we need decentralised writes. Storage: a simple key-value store like DynamoDB or Redis for the short→long mapping works well here. The schema is flat (shortCode → { longUrl, createdAt, expiresAt, clicks }), which is a natural fit for NoSQL. For the redirect path (hot path), I'd put an in-memory cache (Redis) in front of the DB. Most URLs follow a power-law distribution — a small percentage get almost all the traffic — so a cache hit rate of 90%+ is realistic. Click tracking would be async: the redirect service fires a message to Kafka, and a consumer batch-updates the counter in a separate analytics store to avoid write contention on the main DB. Expiry can be handled by a TTL on the cache entry plus a background job that purges expired records from the DB daily. One tradeoff I'm making: custom aliases require a uniqueness check before write, which adds a DB round-trip. Under extreme write load this could be a bottleneck, but given that custom aliases are a minority of requests I'd accept that tradeoff.
