AI Feedback

Strengths

  • Correctly identified both functional and non-functional requirements before jumping into the design.
  • Chose Base62 encoding for alias generation — a well-reasoned, standard approach.
  • Included a CDN layer for redirect latency, showing awareness of read-heavy workloads.
  • Clearly articulated the SQL vs. NoSQL tradeoff and justified the choice.

Areas for improvement

  • Scalability of the ID generation service was not addressed — single point of failure under high write load.
  • No mention of a cache eviction policy (e.g. LRU) for the redirect cache.
  • Rate limiting on the write endpoint was missing, leaving the system open to abuse.
  • Could go deeper on replication strategy and failover for the primary database.

Guiding questions

Does the design handle 100K writes/day and a 10:1 read ratio without bottlenecks?
Partially. The candidate correctly separated the read and write paths and added caching for redirects, which addresses the read load well. However, the write path relies on a single ID generation service with no discussion of sharding or distributed ID generation (e.g. Snowflake IDs), which becomes a bottleneck at scale.
Is high availability addressed for the redirection flow?
Yes. The candidate placed a load balancer in front of stateless redirect servers and included a CDN, ensuring the hot path can survive individual node failures. Database replication was mentioned but not designed in detail.
Are the tradeoffs of key design choices clearly explained?
Mostly. The SQL vs. NoSQL decision was well-justified with reference to the data model. The choice of Base62 over hashing was explained. The candidate did not discuss tradeoffs around TTL storage or consistency vs. availability for the cache layer.

Candidate's reasoning

I chose a relational database (PostgreSQL) because the data model is simple and highly relational — a short code maps to exactly one long URL, and we want ACID guarantees for writes. For reads, I placed a Redis cache in front of the database keyed by the short code, since 90% of traffic will be redirects and these are extremely cache-friendly. Aliases are generated using Base62 encoding of an auto-incrementing ID, which gives us ~57 billion unique 6-character codes. I added a CDN at the edge to serve cached redirects with sub-millisecond latency for popular links. The write path is intentionally simple: accept URL, generate alias, persist, return alias. The main tradeoff I acknowledged is that Base62 + auto-increment leaks information about write volume; a hash-based approach would be more opaque but risks collisions.
