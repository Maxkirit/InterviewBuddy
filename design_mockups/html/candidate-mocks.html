Question

Design a URL Shortener

Design a system like bit.ly that takes a long URL and returns a short alias. The service should redirect users to the original URL when the alias is visited. Consider scale, reliability, and latency in your design.

Requirements

Functional

  • Given a long URL, generate a unique short alias
  • Redirect users from short URL to original URL
  • Short links expire after a configurable TTL
  • Users can optionally choose a custom alias

Non-functional

  • High availability — redirection must never be down
  • Low latency — redirects under 10 ms
  • System should handle 100K writes/day
  • 10:1 read-to-write ratio

AI Feedback

Strengths

  • Correctly chose a token bucket algorithm and justified it over fixed window counting.
  • Redis as the backing store is well-reasoned — fast, atomic increment operations.
  • Considered both per-user and per-IP limiting scopes.

Areas for improvement

  • No discussion of what happens when Redis is unavailable — fail open or fail closed?
  • Distributed rate limiting across multiple nodes was not addressed.
  • Response headers (X-RateLimit-*) and client communication were missing.

Candidate's reasoning

I chose a token bucket algorithm because it allows short bursts while enforcing a sustained rate limit, which is more user-friendly than a fixed window approach. Redis handles the token state with atomic DECR operations, keeping the check-and-decrement consistent under concurrent load. The API Gateway sits in front and delegates the limit check to a dedicated Rate Limiter service so the logic is centralised and reusable across endpoints.
